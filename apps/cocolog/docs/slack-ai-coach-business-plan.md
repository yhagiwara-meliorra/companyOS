# SlackコミュニケーションAIコーチ 事業計画書

## 0. 事業概要

SlackコミュニケーションAIコーチは、Slack上のコミュニケーション行動を分析し、個人およびチームに対して改善提案を返すPLG型B2B SaaSである。主な価値は、Slack内で完結する軽量な導入体験、本文保存を前提としないプライバシー重視の設計、そして個人の行動変容に特化したAIコーチングにある。

本プロダクトは、ITスタートアップやSaaS企業、リモートワーク中心の組織において、Slackを主要な業務OSとして活用する環境を対象とする。1人開発でも運用可能なAPI駆動・自動化中心の構造により、高粗利のスケーラブルな事業として成立を目指す。

## 1. 市場分析

### 1.1 市場カテゴリ

- カテゴリ: AI × Employee Experience × Slackアプリ
- サブカテゴリ: コミュニケーション分析 / チームコーチング / パーソナルAIコーチ
- タイプ: PLG型B2B SaaS

### 1.2 市場サイズ

- Slack利用企業数: 約75万社（有料プラン）
- うち中小〜中堅企業（50〜500名規模）: 約30万社
- コミュニケーション改善・エンゲージメント市場規模: 約30億ドル（2026年予測）
- Slackアプリ市場（推定TAM）: 約5億ドル
- 1%シェア獲得時の売上ポテンシャル: 年間約5億円規模

### 1.3 なぜ今成立するか

- Slackが社内OSとして定着し、AI連携が急速に普及している
- リモート・ハイブリッドワークにより、非言語的マネジメントが難しくなっている
- 個人のコミュニケーション習慣を可視化したい需要が高まっている
- ClaudeやGPTなどのLLM APIが安価かつ高速になり、1人開発でも高度な分析機能を提供できる

### 1.4 競合分析

| 競合 | 特徴 | 弱点 |
| --- | --- | --- |
| Humu | Google出身者による行動科学ベース | 導入コストが高く、Slack連携が弱い |
| CultureAmp | 組織サーベイ中心 | 個人レベルの行動変容に弱い |
| Officevibe | マネージャー向け | 個人の習慣改善には非対応 |
| Lattice | 人事評価・OKR中心 | Slack内で完結しない |
| BetterUp | 1on1コーチング | 高価格でスケールしにくい |

## 3. プロダクトの差別化

| 項目 | 競合 | 本プロダクト |
| --- | --- | --- |
| 導入コスト | 高額な年契約が中心 | Slackインストールのみ |
| データ扱い | メッセージ保存あり | 本文保存なし、統計のみ保持 |
| 分析単位 | 組織単位 | 個人単位＋匿名組織統計 |
| UX | Webダッシュボード中心 | Slack DM内で完結 |
| AI活用 | 限定的 | Claudeによる自然言語分析 |
| 価格 | $5〜$20 / seat | ¥900〜¥1200 / seat |

### 3.1 勝てる理由

- Slack内完結の軽量UXにより導入ハードルが低い
- データプライバシーを重視し、本文保存なしの設計で安心感を出せる
- 組織評価ではなく自己改善にフォーカスしており、使う理由が明確
- API駆動・自動化中心で、少人数でも維持しやすい

## 4. PLG成長戦略

### 4.1 フェーズ1: 最初の100社（0→PMF）

- Slack App Directoryに掲載
- Product Huntでローンチ
- Xやnoteで「Slack習慣AIコーチ」事例を発信
- 無料プランで個人利用を促進し、週次レポートを提供
- 口コミ・紹介によりチーム導入へ拡張

### 4.2 フェーズ2: 100→1000社（PMF→スケール）

- 個人利用者からチーム導入へつなぐバイラル設計
- Slack内に「Invite teammates」CTAを設置
- チームレポート機能でマネージャー導入を促進
- 自動オンボーディングとStripe決済でセルフサーブを完結

### 4.3 バイラル設計

- DM内に「チーム全体の傾向を見てみる」ボタンを設置
- チーム導入時に無料席追加キャンペーンを実施
- Slackステータス連携のような共有導線を用意する
- 例: 「今週のAIコーチスコア: 82」

## 9. 最大のリスクと対策

| リスク | 内容 | 対策 |
| --- | --- | --- |
| Slack API制限 | メッセージ解析制限 | 統計のみ取得し、キャッシュ設計で補完 |
| データプライバシー懸念 | 企業がAI分析を嫌う可能性 | 本文保存なし設計を前面に明示 |
| バイラル鈍化 | PLG拡散が止まる可能性 | Slack内リファラル機能を強化 |
| 開発リソース不足 | 1人開発の限界 | 自動化・ノーコード運用・外注最小限で対応 |
| LLMコスト上昇 | API価格変動 | キャッシュとバッチ処理を最適化 |
