# Handoff Rules

AI CEO が次の担当へ引き渡す条件を定義する。

## 1. AI PM へ渡す条件

以下のいずれかで handoff する。

- `decision = Go` または `decision = Hold` で、要件整理・優先順位付けが必要
- MVP 範囲と非対象範囲が確定した
- 次の実装タスク分解が必要

### 渡すもの

- 課題定義サマリー
- MVP 範囲 / 非対象範囲
- 初期KPI
- `next_actions`

## 2. AI Architect へ渡す条件

以下のいずれかで handoff する。

- 技術構成の選定が必要
- agent / graph 設計が必要
- human approval をどこに入れるか確定が必要
- データ設計・ログ設計・セキュリティ設計が必要

### 渡すもの

- solution_policy
- implementation
- risks.implementation
- approval points

## 3. AI Growth へ渡す条件

以下のいずれかで handoff する。

- 初期顧客獲得方針の設計が必要
- 価格仮説と提案メッセージの検証が必要
- LP / 営業導線 / PoC 導線の設計が必要

### 渡すもの

- target user / pain / urgency
- business_model
- gtm仮説
- 検証すべきメッセージ

## 4. AI Legal へ渡す条件

以下に1つでも該当したら handoff 必須。

- 新しいサービスを追加する
- 新しい個人情報を取得する
- 新しい外部AI / 外部委託先を追加する
- 新料金 / 新プラン / 自動更新 / B2Cセルフサーブを開始する
- API提供を開始する
- 規約・ポリシー・申込み画面の表示変更が必要になる
- 外部公開文書の大幅改訂がある
- 顧客への重要な対外約束が発生する

### 渡すもの

- legal_policy セクション
- 対象データフロー
- 変更対象の規約・ポリシー候補
- 必要契約一覧

## 5. Human Approval へのエスカレーション

以下のいずれかに該当する場合、人間承認を必須化する。

1. 支払い金額が10万円以上変わる可能性のある変更
2. CEO AI の設計変更
3. 外部公開文書の大幅改訂
4. 個人情報・要配慮情報・機密情報の新規取扱い
5. 大きなブランド毀損、炎上、規制対応の可能性
6. 顧客への重要な対外約束
7. 倫理的に境界が曖昧な案件
